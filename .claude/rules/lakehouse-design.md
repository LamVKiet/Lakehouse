# Lakehouse Design Philosophy

## Bronze Layer — Raw Landing Zone

### Characteristics

- **Source of truth**: Raw data exactly as it arrives from source systems.
- **Minimal transformation**: Data stored in its original format, converted to Delta for auditability. Only metadata columns added (`report_date`, `report_month`, `_loaded_at`).
- **Immutable**: Data is **append-only**, never updated or deleted.
- **Schema-on-read**: Flexible schema handling for diverse source systems.
- **Audit trail**: Full history preserved — every version of every record is appended, enabling time-travel and lineage tracking.

### ETL Process

1. **Extract** — Incremental read from MySQL via `read_mysql_incremental()`:
   `WHERE DATE(COALESCE(updated_at, created_at)) = 'YYYY-MM-DD'`
2. **Light Transform** — Add metadata columns by load time: `report_date`, `report_month`, `_loaded_at`.
3. **Load** — Append-only into Delta Lake. **Never use MERGE or UPDATE on Bronze**.

### Best Practices

- Preserve all source data fields for complete auditability (including `is_current` from SCD2 sources).
- Implement incremental ingestion to avoid full reprocessing.
- Partition by `report_date` (yyyyMMdd) for efficient data management.
- Document data sources and ingestion schedules.
- SCD2 logic (close old records, insert new) belongs in Silver, not Bronze.

## Silver Layer — 70 / 30 split

### 70% — Ad-hoc & Depth (Granularity First)

- Flatten all nested structures (e.g. `raw_metadata` JSON → individual typed columns).
- Keep the highest granularity possible — one row per atomic business event, never pre-aggregated.
- Build "wide" tables: pull in all dimensions needed to answer deep analytical questions
  without having to join back to Bronze.
- Internal structure follows **star-schema**: Silver itself acts as a fact table,
  with dimension values embedded as columns (denormalized), not as separate dim tables.

### 30% — Performance, Scale & Traceability (Smart Key)

Apply the **Smart Key** design philosophy to any generated Primary / Unique key column
(i.e. columns named `*_id` that are produced by the pipeline, not source UUIDs carried
from upstream — source UUIDs are exempt and kept as-is for traceability).

**Smart Key format:** `[Time_Component]_[Entity/Branch_id]_[Sequence/Type]`

Example for a Silver event key:
```
20260416_user1205_6   →  log_date(YYYYMMDD) + user_id + event_id
```

**Why Smart Key over UUID:**
- Human-readable: Data Team can debug by eyeballing a key, no lookup needed.
- Naturally aligned with partition pruning: the time component is the partition prefix,
  so Spark/Trino skips irrelevant partitions without a separate filter.
- Sortable & clusterable: keys sort in business order (time → entity → type),
  enabling efficient range scans at billion-row scale.

**Synchronize Smart Key with storage strategy:**
- Partition column = time component embedded in the key (e.g. `log_date`).
- If the engine supports clustering/Z-ordering, cluster on the entity component (e.g. `user_id`).
- This alignment ensures Data Pruning eliminates irrelevant files before any row is scanned.

> **Scope note:** This rule applies only to keys *generated by the pipeline*.
> Source UUIDs (e.g. `event_uuid` carried from Bronze) are exempt — keep them for
> end-to-end traceability back to the raw event.

## Gold Layer — 70 / 30 split

### 70% — Efficiency & Vision (Pre-aggregation)

- Collapse Silver fact rows into topic-focused aggregated tables or Data Marts.
- Pre-calculate all metrics that are expensive at query time:
  compound rates, rolling windows, weighted averages, funnel conversion steps.
- Target: dashboard queries complete in under 3 seconds with zero runtime joins.
- Write strategy: `replaceWhere` overwrite on T-1 partition — idempotent, safe to re-run.

### 30% — Usability & Self-service

- Column naming: use plain business terms, not technical abbreviations
  (`total_revenue_vnd` not `rev_sum`, `unique_buyers` not `cnt_distinct_uid`).
- Every Gold table must be self-contained: a non-technical user should be able to
  drag-and-drop columns into a BI tool and get a correct chart without writing JOIN logic.
- Metrics and dimensions in the same table where they belong together — do not split
  a mart into multiple tables just for normalization if it forces the DA to JOIN.
- Document the business question each table answers in a comment at the top of
  `batch_gold_aggregate.py` and in `analytics/queries/`.
